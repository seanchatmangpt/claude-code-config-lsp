# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

`claude-code-config-lsp` is a Language Server Protocol (LSP 3.18) implementation for Claude Code configuration files. It provides real-time diagnostics, hover documentation, completion, and semantic tokens for settings.json, CLAUDE.md, agents, skills, hooks, marketplace.json, plugin.json, and related configuration files.

The server is **generated** from an RDF ontology (`schema/claude-code-config.ttl`) using the `ggen` tool and the `lsp-max` pack. The ontology is the source of truth — Rust source files are generated artifacts, not hand-edited code.

## Architecture

### Five Configuration Surfaces

The server monitors and validates five distinct config "laws":

1. **Settings** — `settings.json`, `settings.local.json`; runtime permissions, model selection, tool access, hook wiring
2. **Marketplace** — `marketplace.json`; available plugin capabilities
3. **Plugins** — `plugin.json`; plugin declarations (LSP servers, skills, agents, hooks, MCP servers)
4. **Agent definitions** — `agents/*.md`, `AGENTS.md`; agent frontmatter and behavior specifications
5. **Skills** — `skills/*/SKILL.md`; skill metadata and documentation

Additionally: `CLAUDE.md` (project-level instructions), hooks (`.sh` files under `hooks/`), and TOML files (e.g., `.lsp-max-auto.toml`).

### Ontology → Code Generation Pipeline

```
schema/claude-code-config.ttl (RDF source of truth)
  ↓ (ggen reads ontology + SPARQL queries)
  ├→ sparql/*.rq (SPARQL queries extracting rows from ontology)
  ├→ Tera templates (lsp-max pack)
  └→ Generates:
     ├─ src/file_types.rs — file pattern matching & language IDs
     ├─ src/schema.rs — schema field definitions
     ├─ src/hover.rs — hover documentation tables
     ├─ src/completion.rs — completion item tables
     ├─ src/semantic_tokens.rs — token type/modifier definitions
     ├─ src/analyzers/mod.rs — analyzer module registry
     ├─ src/analyzers/{json,claude_md,frontmatter,hook,toml}.rs — analyzer implementations
     ├─ src/declare_model.rs — Declare constraints (process mining)
     ├─ src/virtual_docs.rs — virtual documentation URIs
     ├─ src/capabilities.rs — LSP server capability declarations
     ├─ src/conformance.rs — conformance scoring rules
     ├─ Cargo.toml — manifest metadata
     └─ src/{main,lib,backend}.rs — server entrypoint & backends
```

When the ontology changes:

```bash
ggen sync
```

Regenerates all affected files. Generated files are first-class source and must be inspected and committed. They are NOT read-only or marked with `DO NOT EDIT` banners.

### Backend Handler Flow

**File Open/Change:**
1. Client sends `textDocument/didOpen` or `textDocument/didChange`
2. `Backend::scan_uri_classified()` determines file type
3. Routes to appropriate analyzer(s) based on file classification
4. Analyzer produces diagnostics
5. Server publishes diagnostics via `textDocument/publishDiagnostics`

**Hover/Completion/Semantic Tokens:**
- Requested via LSP protocol (`hover`, `completion`, `semanticTokens/full`)
- Served from lookup tables in `src/hover.rs`, `src/completion.rs`, `src/semantic_tokens.rs`
- Tables are generated from ontology fields

### Analyzers

Each analyzer implements the same trait: read file content, emit findings.

**Built-in analyzers:**
- **claude_md** — Validates `CLAUDE.md` and `agents.md` structure (project vs. global scope detection)
- **frontmatter** — Validates agent and skill frontmatter (`---` YAML blocks)
- **json** — Validates `settings.json`, `plugin.json`, `marketplace.json`, `mcp.json`, `keybindings.json`
- **toml** — Validates TOML config files
- **hook** — Validates shell scripts in `hooks/` directory

Analyzers are instantiated in `Backend::scan_uri_classified()` and run via the `ReplayableAnalyzer` trait (enables resumable/debuggable processing).

### File Classification

`Backend::classify_uri()` determines which analyzer(s) apply:
- Pattern matching on URI (e.g., `skill.md` → "skill" class)
- Patterns defined in ontology; code generated by `file_types.rq` + template

### Scope Detection

The server distinguishes **global** (user's `~/.claude/`) from **project** scope via `is_global_scope()`. Diagnostics are emitted with `source: "claude-code-config-lsp"` (project) or `source: "claude-code-config-lsp[global]"` (global).

### Event Logging (OCEL)

The `OcelAccumulator` records activity events (file opens, scans, diagnostics) in OCEL (Object-Centric Event Log) format. Events are serialized via `emit_ocel_event()`. Disabled by default; use for audit trails or process mining.

## Editing Workflow

### To Add a New Config Surface

Never edit generated files directly. The ontology is the single source of truth.

1. **Edit `schema/claude-code-config.ttl`:**
   - Add a new `ccc:ConfigSurface` individual with file pattern, language ID, and analyzer module reference
   - Optionally add `ccc:SchemaField`, `ccc:DiagnosticRule`, or `ccc:DeclareConstraint` individuals

2. **Run `ggen sync`:**
   ```bash
   ggen sync
   ```
   Generated files update automatically.

3. **Implement analyzer (if needed):**
   - If using an existing analyzer module (e.g., `json`), no code change needed
   - If creating a new module, implement the analyzer stub generated by `ggen`

4. **Commit:**
   - Stage the ontology, regenerated source files, and any analyzer implementations
   - The ontology + generated files form a coherent snapshot

### To Add a New Diagnostic Rule

1. Add a `ccc:DiagnosticRule` individual to the ontology
2. Reference it in the corresponding analyzer's implementation
3. Run `ggen sync` to regenerate table-driven diagnostic code
4. Test via LSP hover/completion to verify the rule appears

### To Update Hover or Completion

Edit the `rdfs:comment` (for hover) or completion table in the ontology. Run `ggen sync` — tables regenerate automatically.
